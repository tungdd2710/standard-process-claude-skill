# AI Model Evaluation

Run every step before committing to a model change in production.

## Step 1 — Catalog freshness (FIRST)

3-way diff before any bench run — never bench a stale quantization:

```bash
# What's on your local/self-hosted instance
# Check your model host's CLI (e.g. `ollama list`)

# What's published
# Check the model provider's library page manually for latest tags

# What's on OpenRouter
curl -s https://openrouter.ai/api/v1/models | jq '[.data[].id]' | grep -i "<model-family>"
```

Pull missing latest tags before benching. If your host is missing a newer tag → pull it first, then bench.

## Step 2 — Candidate slate

Bench ≥3 candidates — always:
- **Open-source self-host** — local inference (Ollama, vLLM, llama.cpp, etc.)
- **Hosted commercial** — Claude / GPT / Gemini via API
- **Cheap/fast option** — a smaller/cheaper model in the same task category

Open-source wins ties within: 0.05 Pearson / 5% accuracy / 50ms p95 latency / 10pp QWK.

Always use LATEST variant. Verify on developer website + OpenRouter API + recent AI news. Never rely on training-data recall for model versions.

## Step 3 — Jurisdiction gate

Check origin for each candidate:
- **Self-hosted** → always OK regardless of model origin
- **Free-tier from jurisdiction-restricted endpoint** → bench-only, NOT production
- **Paid restricted endpoint** → evaluate against your compliance requirements; remove from slate if non-compliant

## Step 4 — Dataset license gate

For every eval dataset:
- Generated by Claude/GPT/Gemini API → **REFUSE** (AUPs are transitive; training/eval on LLM-distilled data likely violates ToS)
- CC-BY / Apache-2.0 / MIT → OK
- GPL / CC-NC / no license → CONCEPT-ONLY (eval insights OK, can't redistribute)
- Log verdict in your project's dataset registry

## Step 5 — Run bench

Use your project's evaluation harness on representative data. Label every result:

| Label | Meaning |
|-------|---------|
| `[N=X SELF-TEST]` | Your measurement on N representative samples |
| `[VENDOR-CLAIMED]` | From vendor docs/benchmarks — never load-bearing alone |
| `[VERIFIED-PROD]` | Confirmed against live production traffic |

Never ship `[VENDOR-CLAIMED]` as the sole justification for a production model choice.

Population note: flag results from non-representative populations; note any calibration gaps on your actual user data.

## Step 6 — P-Vault threshold

If a model scores <50% on the canonical task metric:
1. One re-audit: different prompt, adjusted temperature
2. If still <50% → vault for this task only (not globally)
3. Record vaulting decision with: task slug, date, metric score, reason

## Step 7 — Update spec card + fallback

Every prod model change requires:
- Update your AI feature's spec card/registry entry:
  - `multiModelBenchStatus`: `benched_N=X` or `pending`
  - `modelId`: new winner
  - `fallbackModelId`: previous winner (never drop — always keep a fallback)
- Update relevant environment variables on prod
